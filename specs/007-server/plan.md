## Summary

ArgusAI Server introduces a central service layer using a **result sync model**: test execution remains local (Docker on developer machines), and results asynchronously sync to a central server. This enables team-wide visibility, shared diagnostics, and centralized notifications while preserving the local-first guarantee.

The implementation has three pillars:
1. **Drizzle ORM migration** — Replace direct `better-sqlite3` calls with Drizzle ORM for transparent SQLite/MySQL/PostgreSQL support
2. **Sync infrastructure** — Local sync queue + RemoteHistoryStore decorator + server-side sync API
3. **Server package** — New `packages/server/` with REST API, team management, 企微 notifications, and standalone Dashboard

**Language/Version**: TypeScript 5.x (strict mode), Node.js 20+

**Target Platform**: Linux server (production), macOS (development)  
**Project Type**: Monorepo (pnpm workspaces) — adding `packages/server/`  
**Performance Goals**: 100 concurrent sync requests, <2s API response for 10K runs, <30s sync latency  
**Constraints**: <200MB Docker image, <5% sync overhead on local execution, zero regression in local-only mode  
**Scale/Scope**: 10-50 teams, 100+ developers, 10K+ runs/project

## Implementation Phases

### Phase P1 — Core Infrastructure (Plan Items 1–25 → Tasks T001–T040)

**Goal**: Drizzle ORM migration + sync pipeline + server REST API + auth

#### P1.1 — Drizzle ORM Migration (estimated: 3-4 days)

1. **Add Drizzle dependencies to `packages/core/`**
   - `drizzle-orm`, `drizzle-kit`
   - Keep `better-sqlite3` as the SQLite driver (Drizzle supports it natively)
   - Add `pg` (node-postgres) as optional peer dependency
   - Add `mysql2` as optional peer dependency

2. **Create dialect-specific schema files**
   - `schema-sqlite.ts`: Mirror existing table structure from `migrations.ts` v1+v2
   - `schema-pg.ts`: Same structure with PostgreSQL types (`uuid`, `timestamp`, `serial`)
   - `schema-mysql.ts`: Same structure with MySQL types
   - All schemas add new server columns: `team_id`, `synced_at`, `source_developer`

3. **Create `createDb()` factory**
   - Accepts `DbConfig` with `dialect`, `connectionString`, or `filePath`
   - Returns typed Drizzle instance for the appropriate dialect
   - Local mode: `drizzle(new Database(path))` with SQLite schema
   - Server mode: `drizzle(pgPool)` or `drizzle(mysqlPool)` with appropriate schema

4. **Implement `DrizzleHistoryStore`**
   - Implements existing `HistoryStore` interface
   - Uses Drizzle's query API (dialect-agnostic)
   - Drop-in replacement for `SQLiteHistoryStore`

5. **Implement `DrizzleKnowledgeStore`**
   - Implements existing `KnowledgeStore` interface
   - Uses Drizzle's query API
   - Drop-in replacement for `SQLiteKnowledgeStore`

6. **Migration bridge (v2 → v3)**
   - New migration that adds server-related columns to existing tables
   - Uses `ALTER TABLE ... ADD COLUMN` with defaults (backward-compatible)
   - Detects existing `user_version` and skips table creation if ≥ 2

7. **Update `createHistoryStore()` factory**
   - New path: if Drizzle config available, create `DrizzleHistoryStore`
   - Default path (no config): create `DrizzleHistoryStore` with SQLite (replaces direct `better-sqlite3`)
   - Fallback path: `MemoryHistoryStore` on failure

8. **Verify all existing tests pass with Drizzle backend**
   - Run full test suite — should be green since interface is unchanged
   - Add specific migration tests for v2 → v3 upgrade

#### P1.2 — Sync Infrastructure (estimated: 3-4 days)

9. **Create `SyncQueue`**
   - Uses `sync_queue` table in local SQLite
   - Methods: `enqueue()`, `dequeue()`, `markCompleted()`, `markFailed()`, `getPending()`
   - Exponential backoff: 5s base, 2x multiplier, 5min max

10. **Create `SyncClient`**
    - HTTP client for server communication
    - Methods: `syncRuns(payload)`, `syncPatterns(payload)`, `ping()`
    - Uses native `fetch` (Node 20+)
    - Adds `X-API-Key` header
    - Timeout: 30 seconds per request
    - Returns typed responses

11. **Create `SyncManager`**
    - Orchestrates sync lifecycle
    - Background timer: check queue every 30 seconds
    - Manual trigger: `sync()` method for CLI `argusai sync`
    - Status reporting: `getStatus()` returns queue depth, last sync time, errors

12. **Create `RemoteHistoryStore`**
    - Decorator wrapping `DrizzleHistoryStore` (local) + `SyncQueue`
    - `saveRun()`: writes locally then enqueues sync payload
    - All read methods: delegate to local store
    - Fully transparent to callers

13. **Extend `e2e.yaml` schema**
    - Add `ServerConfigSchema` to `E2EConfigSchema`
    - Add `ARGUSAI_API_KEY` env var support
    - Update `loadConfig()` to resolve server config

14. **Update `createHistoryStore()` to support RemoteHistoryStore**
    - If `serverConfig` present and `sync !== 'disabled'`: wrap with `RemoteHistoryStore`
    - If `serverConfig` absent: return local-only store (unchanged behavior)

#### P1.3 — Server Package (estimated: 5-6 days)

15. **Scaffold `packages/server/`**
    - `package.json` with dependencies: `fastify`, `drizzle-orm`, `argusai-core`, `pg`, `mysql2`
    - `tsconfig.json` extending root config
    - ESM + strict mode

16. **Server config & DB connection**
    - `config.ts`: Zod schema for server environment config
    - `connection.ts`: Multi-dialect DB connection factory
    - `migrate.ts`: Run Drizzle migrations on startup

17. **Authentication system**
    - `api-key.ts`: `generateApiKey()`, `hashApiKey()`, `validateApiKey()`
    - `auth-hook.ts`: Fastify preHandler that validates `X-API-Key`, injects `teamId` into request
    - Public routes excluded: `/api/health`, `POST /api/teams` (bootstrap)

18. **Team management routes**
    - `POST /api/teams` — Create team, return API key (one-time display)
    - `GET /api/teams` — List teams (admin only, initially no admin role needed)
    - `DELETE /api/teams/:id` — Delete team and all data
    - `POST /api/teams/:id/reset-key` — Reset API key

19. **Sync routes**
    - `POST /api/sync/runs` — Receive test run data, auto-register project
    - `POST /api/sync/patterns` — Receive diagnostic patterns, deduplicate
    - Both endpoints: validate team from API key, extract project name, upsert data

20. **Query routes**
    - `GET /api/projects` — List projects with summary stats (team-scoped)
    - `GET /api/runs` — Paginated run listing (team + project scoped)
    - `GET /api/runs/:id` — Run detail with cases and flaky info
    - `GET /api/runs/compare` — Run comparison
    - Trend endpoints: pass-rate, duration, flaky, failures (reuse core logic)
    - Diagnostic endpoints: patterns, fixes

21. **Server app factory & entry point**
    - `app.ts`: `createServerApp()` — registers all plugins and routes
    - `index.ts`: Reads env config, creates DB, starts server
    - Health check: `GET /api/health`

#### P1.4 — Testing & Integration (estimated: 2 days)

22. **Unit tests for Drizzle stores**
    - Test `DrizzleHistoryStore` against SQLite
    - Test all interface methods match `SQLiteHistoryStore` behavior
    - Migration tests: v2 DB → v3 upgrade

23. **Unit tests for sync infrastructure**
    - `SyncQueue`: enqueue, dequeue, retry logic
    - `SyncClient`: mock HTTP responses
    - `RemoteHistoryStore`: verify local write + enqueue
    - `SyncManager`: timer, manual trigger, error handling

24. **Unit tests for server**
    - Auth hook: valid key, invalid key, missing key
    - Sync routes: create run, idempotent re-sync
    - Team routes: CRUD operations
    - Query routes: pagination, filtering, trends

25. **Integration test: end-to-end sync flow**
    - Start test server with SQLite
    - Configure local instance with server URL
    - Run mock test, verify sync, query via API

### Phase P2 — Collaboration Features (Plan Items 26–35 → Tasks T041–T055)

**Goal**: 企微 notifications + Dashboard standalone + diagnostic sync

#### P2.1 — 企微 Notifications (estimated: 2-3 days)

26. **Notification type system**
    - `NotificationChannel` interface: `send(message): Promise<void>`
    - `NotificationConfig`: webhook URL, triggers (failure, success, digest, newFlaky)
    - `NotificationTrigger`: evaluates sync data against config

27. **WeComNotifier implementation**
    - Markdown message formatting (project name, run summary, failed cases, dashboard link)
    - Rate limiting: max 18/min (under 20/min limit), coalesce within 10s window
    - Error handling: log failures, never block sync

28. **Notification trigger engine**
    - On sync completion: evaluate triggers (failure, success, newFlaky)
    - Daily digest: cron-like scheduler (configurable time)
    - Team config stored in `notification_configs` table

29. **Team notification config API**
    - `PUT /api/teams/:id/notifications` — Update notification config
    - `GET /api/teams/:id/notifications` — Get current config

#### P2.2 — Dashboard Standalone Mode (estimated: 3-4 days)

30. **API client with auth**
    - `packages/dashboard/ui/src/api/client.ts`
    - Axios/fetch wrapper with `X-API-Key` header
    - API key from localStorage or login prompt
    - Base URL from `VITE_API_BASE_URL` env var

31. **Login & team selection UI**
    - `LoginScreen.tsx`: API key input, validate against server
    - `TeamSelector.tsx`: Dropdown for multi-team users
    - Auth state in React context

32. **Multi-project navigation**
    - `ProjectList.tsx`: Grid/list of projects with summary cards
    - Click to enter project → existing Dashboard views
    - Breadcrumb navigation: Team > Project > View

33. **Adapt existing pages for server data**
    - `TrendsPage`: Swap data source from local API to server API (same response format)
    - `RunsPage`: Add "synced by" column
    - Empty state: Guide for configuring sync

34. **Dashboard build configuration**
    - `VITE_API_BASE_URL` for server mode
    - `VITE_AUTH_REQUIRED` flag
    - Build scripts for standalone deployment

#### P2.3 — Diagnostic Pattern Sync (estimated: 1-2 days)

35. **Extend sync payload**
    - Include `FailurePattern[]` and `FixRecord[]` in sync data
    - Server deduplicates patterns by signature + team
    - Aggregate occurrence counts across team members
